## The one distinction that governs everything

Two species of artifact live under `docs/regimes/`, handled oppositely:

- **Reference corpus (FROZEN — never regenerate).** The us_house ground truth the role
  evals score against: `docs/regimes/us-house/reference/E1.lock.json` + its sha256-pinned
  files. Freeze discipline (`role-eval-thresholds.md`): a scorer that finds a defect
  surfaces a FINDING; it never edits a pinned artifact. **No factory agent writes here.**
  Letting a scout overwrite the answer key it is about to be graded against is
  model-verifies-model — the exact failure the calibration harness exists to prevent.
- **Factory output (GENERATED by agents).** Everything from Brazil onward, plus the three
  *not-yet-existing* E1 artifacts below. These are produced by specialists through the
  normal phases.

The three E1 artifacts the calibration run creates are NOT in the frozen lock — they don't
exist yet (that's why scout/surveyor/sampler score NOT_APPLICABLE). Creating them is
additive and does not touch the lock. If a validator later finds a defect in a *pinned*
file, that is a FINDING + founder-superseded lock bump, never an in-place edit.

## Task zero — verify the invocation surface (do this before either workflow)

Reasoning: this runbook asserts command names and paths from the repo's own docs, but
those must be confirmed against the built code, not trusted. Three checks:

1. **The gate, live.** `cargo run -p pipeline --bin epoch-gate -- E2`
   Prints the per-role verdict. This is ground truth for *which* roles are blocking —
   calibrate exactly those. Pure filesystem/hash reads; no local Postgres needed.
2. **The scorer harness.** `cargo test -p pipeline role_evals`
   Confirms `validate_sources` / `validate_survey` / `validate_manifest` run here (they are
   eval functions in `crates/pipeline/src/evals/`, invoked via this test — NOT standalone
   `validate-*` bins). The calibration prompt's acceptance is these two commands, nothing else.
3. **Path spelling (silent-failure trap).** The repo uses BOTH `docs/regimes/us-house/`
   (hyphen: the old flat survey `us-house.md` and `reference/`) AND `docs/regimes/us_house/`
   (underscore: the wanted `sources.yaml`, `AUTHORITY.md` per `role-eval-thresholds.md` and
   `CLAUDE.md`). Confirm which spelling each scorer opens. A hyphen/underscore mismatch makes
   the surveyor produce a valid file the scorer never finds → role stays NOT_APPLICABLE with
   no error. Fix the spelling to match the scorer before running, or the calibration silently
   never converges.

If any command name differs, adjust the prompt blocks below to the real names.

## Workflow 1 — calibration run (us_house; unblocks E2)

Produces the three missing E1 reference artifacts through the specialist agents, each gated
by the auditor pass per `source-exploration.md`, WITHOUT editing any frozen file.

Prompt (one-off; paste into an interactive `claude` session in its own worktree):

```
Bring the three NOT_APPLICABLE E1 roles (scout, surveyor, sampler) to threshold by
      having each specialist agent produce its us_house artifact through the normal
      source-exploration phases. Do NOT edit any file under
      docs/regimes/us-house/reference/ (frozen; findings only).
        - scout   -> docs/regimes/us_house/sources.yaml
        - surveyor-> docs/regimes/us_house/AUTHORITY.md on the {url,file} evidence schema
                     (do NOT grandfather the pre-existing docs/regimes/us-house.md)
        - sampler -> a sampler-attributed capture manifest (captured_by = sampler)
      Stage each artifact to a scratch path; promote into docs/regimes/ only after its
      scorer passes. Auditor pass between produce and promote.
      DONE when: cargo test -p pipeline role_evals passes AND
                 cargo run -p pipeline --bin epoch-gate -- E2 exits 0.
```

Why calibration precedes Brazil: E2 is the first *fully factory-driven* epoch on a
non-English regime where output can't be eyeballed. Proving scout/surveyor/sampler against
the one corpus with known answers is what makes their unsupervised Brazil run trustworthy.
Fail-closed on unmeasured roles is correct — opening E2 on never-scored agents is flying
blind on the epoch that most needs the instruments calibrated.

## Workflow 2 — standing factory (Brazil onward; fully autonomous)

Runs only after Workflow 1 turns the E2 gate green.

Standing driver: `GOVFOLIO_LANES=N ./agents/run-loop.sh` — factory lanes execute
`agents/workflows/factory-lane.md`; the prompt below is what each lane iteration
effectively does (kept for single-session/manual runs):

```
Run the coverage factory. Repeatedly: select the highest priority_score jurisdiction
      in the current epoch with coverage_phase < live and no live lease (orchestration
      step 2d); lease it via `cargo run -p worker --bin jurisdiction-lease -- claim`
      (atomic, goal 097 — never raw claimed_by/claimed_at writes); execute its current phase with the mapped
      specialist agent + its allocator-assigned skills + the source SAF; stage the artifact,
      run the auditor pass where mandated, validate, commit locally without JOURNAL, submit
      an immutable receipt, and wait. Never advance/block/release phase, push, or merge.
      Drive ALL work through registry coverage_phase transitions ONLY — never write goal
      files (invariant 9: unlisted goal files are quarantined). Honor the epoch gate before
      entering each new epoch. Continue until every jurisdiction in the epoch reaches
      coverage_phase = live or blocked:<reason>.
```

This is the discovery → artifact → planning automation, per source: scout (sources.yaml) →
surveyor (AUTHORITY.md) → sampler (fixtures + manifest) → spec-writer + test-designer
(plan + draft expected outputs) → builder (conformance-green adapter), auditor between,
Sentinel WATCH after live. The registry row is the unit of work; no per-jurisdiction goal
file is ever created — that is what "no hand-written goals from E2" means, and it is what
keeps the factory compatible with the goal-queue integrity invariant.

## Stage-then-promote (the rule both workflows assume)

Recommended default, and why: you run fully unattended, so a mid-run failure must never
leave a half-written or hallucinated artifact in `docs/regimes/` where a *later* epoch
could treat it as authoritative. Agents write to a scratch location (e.g.
`target/factory-staging/<jurisdiction>/<phase>/`); an artifact promotes into `docs/regimes/`
only after its scorer passes and the auditor pass (where mandated) is green. Same
fail-closed/atomic philosophy as the migration and tf-plan guardrails.

If the current factory code writes in place instead, wiring stage-then-promote is a small
precondition task — do it before the first unattended Brazil run, not after a partial
artifact has already landed.

## Monitoring & stop conditions

- Semantic heartbeat: `./agents/monitor.sh`, `jurisdiction-lease status`, and
  `govfolio-loop receipt-status <receipt-id>`. A phase may advance only with an applied
  receipt for the exact green source commit; pending rows are integrator-owned.
- Gate re-check anytime: `cargo run -p pipeline --bin epoch-gate -- E2` (or the next epoch id).
- Blocked jurisdictions are not failures — `blocked:<reason>` rows are transparency-scorecard
  content. Review them, don't force them.
